# Dataset Builder

The dataset builder is the interactive tool that turns a live webcam feed into labeled training files for the gesture pipeline. It sits at the boundary between a human signer and everything downstream—training scripts, evaluation, and export—so its job is not only to record pixels but to produce **consistent numeric features**, **traceable metadata**, and **repeatable capture sessions** that you can reason about later.

You run it from the `ai_transcript` directory with `python dataset_builder.py`. On startup it loads the project label specification so static letters and motion labels are validated against the same canonical names the rest of the code expects. The main menu lets you capture static alphabet samples, capture motion sequences, or exit. Before each capture you can enter a signer identifier (for example `p01`) so splits and audits can treat signers as first-class entities. You are also prompted for session context—location, camera type, lighting, background, dominant hand, and free-form notes—with the option to reuse the previous session’s answers when you are recording many clips in one sitting. That context is stored alongside the samples so a future you (or a collaborator) can tell *how* the data was collected, not only *what* label it received.

Under the hood the builder uses **MediaPipe Tasks Hand Landmarker** with a downloaded `.task` model stored under `models/`. Static capture uses a single-hand detector; motion capture requests two hands so features can represent both arms in a fixed layout. Features are **wrist-centered** and **scale-normalized in 2D** per hand, producing 63 numbers per hand. For motion, the two hands are sorted left-to-right and concatenated, so each frame is a **126-dimensional** vector. Missing second-hand slots are zero-padded so tensor shapes stay uniform.

**Static** mode is aimed at steady poses for letters A–Z. After a short countdown, the tool looks for stable frames using a sliding variance check on recent landmark vectors, collects stable candidates, and—by default—keeps the **five** frames closest to their centroid in feature space so outliers from micro-movements are dropped. Each saved sample is a `.npy` file containing a single `(63,)` vector. **Motion** mode records **variable-length** sequences stored as `.npz` files with a `seq` array of shape `(T, 126)`. Segmentation is energy-based: movement between consecutive frames is compared to **adaptive** start and keep thresholds estimated from recent idle motion so the pipeline is less sensitive to a fixed sensitivity that only works in one room or lighting setup. A small prebuffer is included when motion starts so the first dynamics of the sign are less likely to be cut off. Sequences are trimmed to drop very low-energy tails, capped to a maximum length per sample, and summarized with quality metrics such as peak and mean motion energy, hand visibility ratios, and a simple QC score.

When a motion segment ends, the builder pauses on a **review** screen. You see a concise diagnostic overlay (frame counts, energies, QC score, two-hand ratio, and similarity to recent captures of the same label) and choose whether to **save**, **retry**, **discard**, or **cancel** the session. Near-duplicate detection compares a compact signature of the sequence against the most recently saved files for that label and warns when cosine similarity is very high, which helps catch accidental repeats without blocking legitimate variation.

Everything you save is wired into the project’s manifest system. Custom webcam captures append rows to `dataset/sample_manifest_custom.jsonl` and session-level rows to `dataset/metadata_custom.jsonl`. Sample records include paths relative to `ai_transcript`, task type, label, signer, session id, and for motion the richer QC block and capture context. That design keeps raw files on disk in predictable folders—**`dataset/raw_custom/static/<LABEL>/`** and **`dataset/raw_custom/motion/<LABEL>/`**—while the JSONL files provide a queryable index for training and auditing.

In short, the dataset builder is the practical front door for growing the custom portion of the corpus: it enforces label hygiene, records who and where the data came from, segments motion in a trainable way, and gives you immediate feedback before a bad clip lands in the dataset.

The recommended workflow is now a simple three-step loop: capture first, clean second, train third. During capture you can enter free-form motion labels as your vocabulary evolves (`python dataset_builder.py`). After you build enough data, run preprocessing to canonicalize labels and produce a clean training manifest (`python preprocess_labels.py --min_motion_samples 3 --drop_missing_files`). Then train from the cleaned manifest so the model only sees finalized labels (`python train.py --mode motion --manifest_path dataset/sample_manifest_cleaned.jsonl`). This keeps collection fast while preserving training consistency.
